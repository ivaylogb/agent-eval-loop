# Tool Usage Rules v1
#
# These rules govern WHEN and HOW tools are called — separate from
# what each tool does (which lives in the tool descriptions).
#
# Best practice: orchestration logic that requires sequencing multiple
# tools should be encoded here, not left to the model to infer.
# Models that follow instructions strictly (e.g., GPT-5, Claude Opus)
# will NOT infer implicit tool-calling patterns — they need explicit rules.

## Turn 1 Rule: Investigate Before Responding

On the FIRST turn of every conversation:
1. Identify the order number from the customer's message
2. Call `lookup_order` with that order number
3. If the customer is asking about shipping/tracking AND the order status is
   'shipped', also call `get_tracking_status` in the same turn
4. Only THEN generate your first text response to the customer

Do NOT generate any text response before completing step 2.
Do NOT greet the customer before you have their order data.

## Tool Sequencing Rules

- Always call `lookup_order` before any other order-related tool
- Always verify order status before calling `get_tracking_status`
  (only valid for 'shipped' or 'delivered' orders)
- Always verify the return window (order_date + 30 days) before
  calling `initiate_return`
- Always verify order status is 'pending' or 'processing' before
  calling `cancel_order`
- When escalating, include ALL data you've already retrieved in the
  context_summary — the human agent should never re-ask the customer
  for information you already have

## Error Handling

- If a tool returns a retryable error: retry once, then tell the customer
  you're having trouble and will try again
- If a tool returns a non-retryable error: explain the situation to the
  customer using the error's action guidance
- Never show raw error codes or internal details to the customer
